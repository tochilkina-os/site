# План оптимизации сайта для мобильных телефонов

Дата: 2026-05-05

## Контекст

Проект — статический сайт:

- `index.html`
- `css/site.css`
- `js/site.js`

Специального skill для чистого HTML/CSS в доступном списке нет, поэтому план основан на фактической структуре HTML, CSS и JS.

## Проблема

На сайте уже есть базовая адаптивность через media query для `1120px` и `760px`, но мобильная версия остаётся тяжёлой и местами неудобной: крупные отступы, большие карточки и секции, горизонтальные карусели с фиксированными ширинами, длинная галерея, сложный блок мониторинга и тяжёлые декоративные эффекты.

## Затронутые файлы

- `css/site.css`
  - основные мобильные правки в существующих `@media (max-width: 760px)`;
  - добавить более узкий breakpoint `@media (max-width: 480px)`;
  - адаптировать hero, карточки, карусели, мониторинг, галерею, форму, навигацию.
- `index.html`
  - проверить и добавить `loading="lazy"` там, где изображения находятся ниже первого экрана;
  - возможно добавить `decoding="async"` для изображений;
  - главную фотографию оставить без lazy-loading, потому что она находится в первом экране.
- `js/site.js`
  - вероятно без изменений;
  - проверить только поведение мобильного меню, каруселей и glider при resize.

## Корневая причина

CSS в основном построен как desktop-first:

- `.section` имеет `padding: 112px 0`, на мобильных снижено только до `82px`, что всё ещё много для телефона.
- `.hero-title` на мобильных использует до `13vw`, из-за чего заголовок может занимать слишком много экрана.
- `.certificate-carousel`, `.education-report-carousel`, `.monitor-panel`, `.contact-form` имеют крупные padding/radius/shadow, что создаёт тесноту на экранах 320–390px.
- `.carousel-track` на мобильных использует `grid-auto-columns: minmax(250px, 86%)`; при контейнере 288–343px карточки могут быть неудобными и создавать ощущение горизонтального переполнения.
- Галерея `.group-gallery` на телефоне остаётся с высотой `240px` на каждую картинку, из-за чего блок становится очень длинным.
- Декоративные blur/blob/shadow/animation элементы могут ухудшать производительность на слабых мобильных устройствах.

## Предлагаемые изменения

### 1. Общая мобильная сетка

- Уточнить `.container` для телефонов: уменьшить боковой отступ до 16px, на очень узких экранах до 14px.
- Уменьшить вертикальные отступы секций на `max-width: 760px` и ещё сильнее на `max-width: 480px`.

### 2. Hero-блок

- Снизить `padding-top`, `min-height` и размер заголовка на телефонах.
- Сделать портрет компактнее: ограничить `max-width`, уменьшить визуальные эффекты.
- Сохранить текущий порядок: фото сверху, текст ниже, если визуально он остаётся удобным.

### 3. Навигация

- Оставить мобильное меню, но сделать его компактнее:
  - меньше padding;
  - menu panel не должен занимать лишнюю высоту;
  - сохранить touch target 44px+.
- Проверить, что `body.nav-open { overflow: hidden; }` не мешает скроллу при длинном меню.

### 4. Карточки и сетки

- Для `diplom-grid`, `parents-cards`, `four-columns`, `group-gallery` оставить одну колонку на телефонах, но уменьшить padding/card radius.
- У карточек уменьшить min-height, чтобы блоки не выглядели растянутыми.
- Для галереи сделать меньшую высоту строк, например 180–210px на телефонах.

### 5. Карусели сертификатов и отчётов

- На мобильных сделать карточки почти на всю ширину, но без переполнения: `grid-auto-columns: minmax(0, 100%)` или `minmax(240px, 92%)` в зависимости от желаемого UX.
- Уменьшить padding карточек и контейнеров.
- Улучшить `scroll-snap`, чтобы свайп был предсказуемым.
- Проверить, что кнопки карусели не ломают layout.

### 6. Блок мониторинга

- Уменьшить padding `.monitor-panel`.
- Перестроить `.stats-head`, `.summary-main`, `.stat-row` под узкие экраны.
- Для длинных подписей направлений развития разрешить переносы, убрать риск горизонтального переполнения.
- Годовой переключатель оставить горизонтально скроллируемым, но сделать ширины кнопок адаптивнее.

### 7. Форма контакта

- Уменьшить padding формы.
- Проверить размеры input/select/textarea.
- Оставить кнопку на всю ширину — это удобно для мобильных.

### 8. Производительность на мобильных

- Отключить или ослабить тяжёлые декоративные эффекты на телефонах:
  - blur blobs;
  - крупные box-shadow;
  - hover-transform, которые не нужны на touch.
- Для изображений ниже первого экрана добавить/проверить `loading="lazy"` и `decoding="async"`.
- Рассмотреть CSS `content-visibility: auto` для длинных секций ниже первого экрана, если нужна дополнительная оптимизация.

### 9. Доступность

- Сохранить touch targets не меньше 44px.
- Проверить читаемость шрифтов: не опускаться ниже примерно 14–16px для основного текста.
- Убедиться, что контраст и фокус формы не ухудшатся.

## Тест-план

- Проверить в DevTools/браузере ширины:
  - 320px;
  - 360px;
  - 390px;
  - 430px;
  - 768px.
- Проверить, что нет горизонтального скролла страницы.
- Проверить мобильное меню: открытие, закрытие, переход по якорям.
- Проверить карусели сертификатов и отчётов свайпом и кнопками.
- Проверить блок мониторинга при переключении годов.
- Проверить форму контакта.
- Проверить, что изображения ниже первого экрана грузятся lazy.
- Прогнать Lighthouse mobile до/после и сравнить Performance/Accessibility/Best Practices.

## Риски

- Низкий–средний риск: основная работа в CSS, но много блоков на одной странице.
- Главный риск — случайно ухудшить desktop/tablet вид, поэтому изменения лучше ограничить media query.
- Второй риск — карусели: изменение ширины карточек может повлиять на шаг прокрутки в `site.js`, нужно проверить вручную.

## Рекомендуемый порядок внедрения

1. Сначала внести только CSS-правки в mobile media query.
2. Проверить мобильные ширины и отсутствие горизонтального скролла.
3. Добавить лёгкие HTML-оптимизации изображений.
4. Проверить JS-поведение каруселей и переключателя годов.
5. При необходимости точечно поправить `site.js`, но только если после CSS-изменений шаг прокрутки станет некорректным.
